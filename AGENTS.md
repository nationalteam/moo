# Repository Guidelines

## Project Structure & Module Organization
This repository is a small TypeScript web app for finding nearby high-rated Tabelog restaurants.

- `src/server.ts`: Express server, static file hosting, and `/api/restaurants` endpoint.
- `src/scraper.ts`: Tabelog scraping and parsing logic.
- `public/index.html`, `public/app.js`, `public/style.css`: browser UI, map interaction, and styling.
- `dist/`: build output generated by TypeScript (`npm run build`).

Keep backend logic in `src/` and browser-only logic in `public/`. Do not mix scraping code into frontend files.

## Build, Test, and Development Commands
- `npm install`: install dependencies.
- `npm run dev`: start local development server via `tsx` (`http://localhost:3000` by default).
- `npm run build`: compile TypeScript to `dist/` using `tsc`.
- `npm start`: run compiled server from `dist/server.js`.
- `PORT=8080 npm start`: run production build on a custom port.

## Coding Style & Naming Conventions
- Language: TypeScript (backend) and plain JavaScript (frontend).
- Indentation: 2 spaces; keep lines readable and avoid dense one-liners.
- Use `camelCase` for variables/functions, `PascalCase` for interfaces/types (for example, `Restaurant`), and clear verb-based function names (`fetchRestaurants`, `parseCard`).
- Follow existing style: ES modules, double quotes, and semicolons.
- Keep `tsconfig.json` strictness intact (`"strict": true`); avoid `any` unless justified.

## Testing Guidelines
Automated tests are not configured yet (`npm test` currently fails by design). For each change:

- Run `npm run build` to catch type and compile errors.
- Manually verify critical flows in the browser (search, map markers, API error handling).

If you add tests, place them near source as `*.test.ts` or under a dedicated `tests/` folder, and update `npm test` accordingly in the same PR.

## Commit & Pull Request Guidelines
Recent commits use short, purpose-first messages such as `fix: ...` and `docs: ...`. Follow this pattern:

- Prefer `type: concise summary` (`fix:`, `docs:`, `feat:`).
- Keep commits focused on one logical change.
- PRs should include: what changed, why, how it was validated (`npm run build`, manual checks), and screenshots/GIFs for UI updates.
- Link related issues when applicable.
